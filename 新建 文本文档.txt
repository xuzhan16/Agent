•	除最终报告外，尽量统一 JSON
•	字段名称固定
•	缺失字段给默认值
•	方便写回学生 JSON
校验原则
•	输出必须能被程序解析
•	列表就是列表，分数就是数字，结论就是字符串
•	不能让模型自由发挥成不可控长文本

一、我建议你借鉴 SmartResume 的部分是什么
你不需要把 SmartResume 完整搬进来，但有 4 个思想非常值得融入。
1. 先“恢复简历内容”，再“让大模型理解”
这是它最值得借鉴的一点。
SmartResume 不是拿原始 PDF 或 OCR 结果直接丢给模型，而是先尽量把简历中的内容恢复成更接近人类阅读顺序的文本，再交给 LLM 做结构化抽取。README 明确写到它会从 PDF、图片和 Office 文件中提取文本，并通过版面检测重建阅读顺序，然后再做结构化解析。
对你项目的启发
你的 resume_parse 模块不要设计成：
•	文件上传 
•	直接提纯文本 
•	一次 prompt 全解析 
而应该分成：
•	文件解析层 
•	文本/布局整理层 
•	结构化抽取层 
•	状态写回层 
________________________________________
2. 把简历解析做成“pipeline”而不是单步调用
SmartResume 的 Python API 是 analyzer.pipeline(...) 这种形式，还支持指定抽取类型，例如只抽 basic_info、work_experience、education。
对你项目的启发
你可以把 resume_parse 也做成分阶段管线：
•	extract_raw_content 
•	reconstruct_resume_text 
•	parse_resume_sections 
•	normalize_resume_fields 
•	write_state 
这样即使后面某一步效果不理想，也方便替换。
________________________________________
3. 把“抽取字段”拆成模块，而不是一次性全抽
SmartResume 支持按字段类型抽取，这说明它本身就在强调“分块解析”而不是一口气让模型做所有事。
对你项目的启发
你也应该把学生简历解析拆成几块：
•	基本信息 
•	教育背景 
•	实习经历 
•	项目经历 
•	技能证书 
•	求职意向 
这样：
•	prompt 更稳定 
•	JSON 更稳定 
•	出错时更容易定位 
________________________________________
4. 用“版面/结构感知”服务后续能力画像
SmartResume 的亮点不是单纯 OCR，而是 layout-aware parsing，也就是版面感知式解析。
对你项目的启发
你后面不是只要“简历字段”，还要做：
•	学生能力画像 
•	人岗匹配 
•	路径规划 
•	报告生成 
所以简历解析的输出不能只有“字符串字段”，还要尽量保留：
•	段落类别 
•	经历的时间顺序 
•	标题层级 
•	内容来源段落 
这会让后面学生画像更稳。
________________________________________
二、你不应该直接照搬 SmartResume 的部分
SmartResume 很强，但你这个项目目标不一样，所以有些部分你只需要借鉴，不需要完整实现。
1. 不必先做重型 OCR/版面模型体系
SmartResume 强调 layout detection、OCR、PDF metadata 等完整链路。
但你现在项目重点不是“简历解析系统本身”，而是“职业规划智能体”。
所以你的做法应该是
第一版优先：
•	Word/docx 直接解析文本 
•	PDF 先做轻量文本抽取 
•	图片简历作为后续增强 
•	版面恢复先做轻量规则，不上复杂视觉模型 
也就是：
先借思想，不先上重依赖。

2. 不必照它做成独立大项目
SmartResume 是一个完整简历解析系统，甚至支持本地模型部署和较完整的配置体系。
但你现在更适合把它融进你现有的统一接口框架。
所以你的模块定位应是
不是“独立 SmartResume 子项目”，而是：
你系统中的 resume_parse 业务实现模块
它的输出直接写进 student.json，供：
•	student_profile 
•	job_match 
•	career_path_plan 
•	career_report 
继续使用。

________________________________________
第 1 段：文件适配层
目标
统一接收不同格式简历。
输入
•	docx 
•	pdf 
•	txt 
•	图片简历（可后续增强） 
输出

目标
把原始提取结果恢复成“更像人类阅读”的简历文本。
为什么这一步很重要
很多 PDF 简历的问题不是“看不到字”，而是：

•	标题和正文粘连 
•	时间线被打乱 
•	项目/实习混在一起 
SmartResume 的重要价值就在于会做 reading order reconstruction，也就是阅读顺序重建。
你项目里的实现建议
第一版不必做重型布局检测模型，但至少做：
•	标题识别 
•	分段 
•	左右栏常见模式修复 
•	时间格式识别 
•	列表项整理 
输出：
•	resume_blocks 
•	每个 block 带 section_guess 
比如：
•	basic_info 
•	education 
•	internship 
•	project 
•	skills 
•	awards 
•	self_evaluation

第 3 段：分块结构化抽取层
目标

这一步最接近 SmartResume 的“指定 extraction types”思路。README 里给了 extract_types=["basic_info", "work_experience", "education"] 这种方式。

•	basic_info_extract 
•	education_extract 
•	experience_extract 
•	project_extract 
•	skills_extract 
•	intention_extract

好处
•	减少大模型一次处理的信息量 
•	降低字段漏提 
•	JSON 结构更稳

•	project_experience 
•	competition_experience

•	意向城市

这一步做完，后面的 student_profile 和 job_match 会顺很多。

目标
把解析结果真正纳入你的统一系统状态。
建议写回到：

•	实习 
•	证书

•	自我评价 
•	原始段落映射 
这样做的意义
•	basic_info 方便快速展示与索引 
•	resume_parse_result 方便后续深度分析 
这也与你原本“每次调用后更新 student.json”的思路完全一致。

四、如果按 SmartResume 思想改造，你这个模块的核心设计原则
我建议你把简历解析模块定成下面这几条原则。
原则 1：先恢复结构，再做抽取
不是直接整份简历喂模型，而是先尽量恢复段落结构和阅读顺序。
原则 2：分块抽取，别一口气抽完
把基础信息、教育、经历、技能分开处理。
原则 3：抽取之后必须标准化

原则 4：保留原始来源痕迹
后面如果匹配结果有问题，你要能追溯是简历哪一段抽出来的。
原则 5：输出要服务后续模块
你的解析结果不是终点，而是学生画像和人岗匹配的输入。

先说结论
最适合借鉴并融合的
•	SkillNER / SkillNer：适合借鉴“从简历文本中抽显式技能、证书，并做去重/归一”的思路。它明确支持从非结构化文本、岗位文本和简历中抽取 skills and certifications，并用技能知识库做链接以减少重复。(GitHub)
•	ESCO Skill Extractor：适合借鉴“把文本中的技能和职业映射到标准 taxonomy”的思路。它明确是从 CVs 或 job descriptions 中提取 ESCO skills 和 ISCO occupations，并基于 embedding 相似度做匹配。(GitHub)
•	Tabiya Livelihoods Classifier：最适合借鉴“多实体抽取 + 实体链接”的思路。它不是只抽 skill，而是同时抽取 Occupation、Skill、Qualification、Experience、Domain 五类实体，并采用“先抽实体、再做相似度检索链接”的两步法。(GitHub)
•	pyresparser：适合借鉴“用轻量规则先抽基础结构字段”的思路，比如 skills、education、experience、college name 等。它更适合做前置基础层，不适合直接做最终能力画像。(GitHub)

•	不建议直接把 ESCO 全套 taxonomy 原样作为你项目的主标签体系。它很强，但对你这个中文大学生职业规划项目来说太重、太欧标化，首版接入成本高，而且你后面人岗匹配面对的是企业给的中文岗位数据。ESCO 更适合作为“设计启发”，不是首版主干。(GitHub)
•	不建议直接把 SkillNER / ESCO extractor 当成“最终学生画像模块”来用。它们更偏“抽取器”，不负责你项目要求里的学习能力、创新能力、沟通能力、抗压能力、完整度评分、竞争力评分。A13 明确要求这些内容，所以最后仍然需要你自己的“画像归纳层”。(GitHub)

SkillNER 借什么
SkillNER 明确支持从 resumes 和 texts 中抽取 skills and certifications，还强调用技能知识库做链接来防止重复。(GitHub)
ESCO extractor 借什么
ESCO Skill Extractor 的核心是：
把文本抽出来的 skill/occupation 链接到标准 taxonomy，而不是保留一堆自由文本标签。它支持从 CVs 提取 ESCO skills 和 ISCO occupations。(GitHub)
在你项目里怎么融合

1）先抽短语，再归一

•	Python开发
•	Python 编程
•	熟悉 Python
•	掌握 Python
都统一映射成：
•	Python
2）把技能和资格分层存储
分成：
•	hard_skills
•	tool_skills
•	certificate_list
•	qualification_tags
这样后面岗位画像就能和学生画像按层对比。
为什么适合
因为你后面做人岗匹配时，最怕的是：
•	学生端叫“Java开发”
•	岗位端叫“Java”
•	最后没匹配上
所以技能标准化是必须做的。

借什么
Tabiya 的特点不是只看 skill，而是抽：

•	Skill
•	Qualification
•	Experience
•	Domain

为什么它特别适合你
因为你的 student_profile 并不只是做技能表，它还要生成：
•	实践能力
•	方向倾向
•	经历强度
•	发展潜力
•	后面的人岗匹配维度
这和 Tabiya 那种“多实体、多维度”抽取思路非常接近。
在你项目里怎么融合
你可以把 student_profile 的中间特征层设计成：
•	occupation_hints：学生经历暗示的岗位方向
例如前端开发、数据分析、产品助理
•	skills：显式技能
•	qualifications：学历、证书、竞赛
•	experience_tags：项目、实习、科研、社团
•	domain_tags：如教育、金融、电商、数据、GIS
这一步做完之后，再交给 LLM 去做“软能力 + 发展潜力”总结，就会比直接喂简历稳很多。

•	awards
•	skills
•	intention
这一步主要承接你前面的简历解析模块。
________________________________________
第 2 层：做“显式实体抽取与归一”
这里融合 pyresparser + SkillNER + ESCO extractor 的思路。
做什么
•	学历归一
•	专业归一
•	技能归一
•	证书归一
•	工具归一
•	经历标签归一
目标
得到一个标准化中间层：
{
  "degree": "本科",
  "major": "软件工程",
  "hard_skills": ["Python", "Java", "SQL"],
  "tool_skills": ["Git", "Excel", "MySQL"],
  "certificate_list": ["英语六级"],
  "experience_tags": ["项目经历", "实习经历", "竞赛经历"]
}
价值
这一步让后面匹配和评分有稳定输入。
________________________________________
第 3 层：做“多实体职业能力特征建模”
这里融合 Tabiya 的思路。
做什么
基于项目、实习、获奖、校园经历，构建：
•	occupation_hints
•	experience_strength
•	domain_tags
•	qualification_tags
•	practice_evidence
举例
如果学生有：
•	电商数据分析实习
•	Python + SQL 项目
•	数据可视化比赛
那中间特征可以长成：
•	occupation_hints = ["数据分析", "BI分析"]
•	domain_tags = ["电商", "数据分析"]
•	practice_evidence = ["数据清洗", "可视化分析", "业务报表"]
价值
这一步让你后面不只是“比技能”，还能比方向与经验。
________________________________________
第 4 层：调用 LLM 做“隐式能力画像归纳”
这里是你自己的部分，不是开源项目能直接替代的。
输入
前面三层已经整理好的中间特征，而不是原始简历。
让 LLM 归纳
•	学习能力
•	创新能力
•	沟通能力
•	团队协作能力
•	抗压能力
•	发展潜力
•	实践能力总结
为什么必须单独做
因为 A13 里这些能力是刚需，但现有开源抽取器大多不擅长这部分。它们擅长“抽 skill/qualification”，不擅长“把经历翻译成职业素养与潜力”。(GitHub)
________________________________________
第 5 层：规则评分 + 模型解释
这里也不适合完全照搬开源项目。
做什么
•	完整度评分：规则主导
•	竞争力评分：规则打底，模型解释修正
•	strengths / weaknesses / missing_dimensions
原因
A13 要求你输出的是“就业能力画像”，不是抽取报告。评分和解释必须是你项目自己的逻辑。
________________________________________
如果不融合，有没有必要
我认为不建议完全不融合。
理由是：如果你完全不借这些开源思路，你的 student_profile 很容易退化成“把简历再总结一遍”，会有三个问题：
1.	技能标签不稳定
没有 SkillNER / ESCO 类思路时，同义技能容易分裂，后面人岗匹配会飘。(GitHub)
2.	经验维度太单薄
没有 Tabiya 类思路时，你容易只看到 skills，看不到 qualification、experience、domain 这些同样重要的实体。(GitHub)
3.	画像难以标准化复用
没有 taxonomy / entity-linking 思路时，学生画像难以和岗位画像做稳定对齐。(GitHub)
所以我的判断是：
•	有必要融合思路
•	没必要硬接全部项目
•	更没必要照搬整套系统
________________________________________
最终给你的融合版开发思路
你可以把 student_profile 模块概括成下面这条链：
融合版链路
1.	读取 resume_parse_result
承接 SmartResume 风格的结构化简历结果
2.	规则整理基础字段
借 pyresparser 思路，先稳定学历、专业、技能、经历、证书
3.	显式技能与资格归一
借 SkillNER + ESCO extractor 思路，做技能/证书/职业词标准化
4.	多实体中间特征构建
借 Tabiya 思路，构建 occupation / skill / qualification / experience / domain 五类特征
5.	调用统一 LLM 接口做能力归纳
得到软技能、发展潜力、实践能力总结
6.	规则评分 + 模型解释
输出完整度、竞争力、优势、短板、缺失维度
7.	写回 student.json
形成 student_profile_result
________________________________________
一句话结论
这部分最适合融合 SkillNER 的“显式技能抽取与去重”、ESCO extractor 的“标准 taxonomy 映射”、Tabiya 的“多实体抽取与链接”、以及 pyresparser 的“轻量基础字段整理”思路；但最终的软能力归纳、评分逻辑和可匹配画像输出，仍然必须由你自己的 student_profile 模块来完成。
下一步最实用的是把这个融合思路落成一份模块设计。






一、job_profile 模块的详细开发思路
1. 先明确这个模块要解决什么问题
job_profile 不是简单“抽取一条岗位 JD”。
它真正要做的是：
把岗位数据中的岗位名称、岗位描述、公司信息、薪资/行业/城市等信息，转换成可标准化、可聚合、可匹配、可挂图谱的“标准岗位画像”。
也就是说，这个模块最终不是服务于“展示某条招聘信息”，而是服务于：
•	job_match：岗位画像 vs 学生画像 
•	career_path_plan：职业路径规划 
•	career_report：职业规划报告 
•	知识图谱：岗位节点与关系边 
________________________________________
2. 这个模块在你系统中的位置
建议顺序是：
1.	resume_parse 
2.	student_profile 
3.	job_profile 
4.	job_match 
5.	career_path_plan 
6.	career_report 
所以 job_profile 是岗位侧主模块，相当于和 student_profile 对称。
________________________________________
3. 这个模块的输入来源
建议输入不是原始 Excel 全表直接丢给模型，而是来自你前面的数据处理结果。
输入建议包括：
•	清洗后的岗位明细表 
•	岗位标准名映射表 
•	某个标准岗位对应的岗位组数据 
•	岗位描述 job_desc 
•	公司简介 company_desc 
•	公司行业 / 规模 / 企业性质 
•	薪资、城市、更新时间 
•	可选：技能词表、专业词表、证书词表、软技能词表 
关键原则
按“标准岗位名”分组后再做画像，不要按单条 JD 做最终画像。
例如：
•	Java开发工程师 
•	Java工程师 
•	后端Java开发 
如果都映射成标准岗位 Java开发工程师，那画像应该是对这个岗位组做聚合，而不是只看某一条。
________________________________________
4. 这个模块的总体处理链
我建议你把 job_profile 拆成 六层处理链。
________________________________________
第 1 层：岗位组读取与样本整理
目标
从清洗后的岗位数据中，取出某个标准岗位对应的岗位组。
例如
输入标准岗位名：
•	实施工程师 
从岗位表中找到：
•	所有 standard_job_name = 实施工程师 的记录 
输出
一个岗位组数据集，例如：
•	岗位名称列表 
•	若干条岗位描述 
•	若干条公司简介 
•	薪资区间统计 
•	城市分布 
•	行业分布 
设计重点
这一层不要直接喂给模型全部原文，而是先做：
•	抽样 
•	摘要 
•	聚合统计 
________________________________________
第 2 层：显式要求抽取
目标
先用规则/词表把岗位里明确写出来的硬条件抽出来。
建议抽取的字段
•	degree_requirement 
•	major_requirement 
•	experience_requirement 
•	hard_skills 
•	tool_skills 
•	certificate_requirement 
•	practice_requirement 
•	soft_skill_hints 
•	work_mode_tags（出差、驻场、轮班、值班等） 
举例
从岗位描述中抽：
•	本科及以上 
•	计算机相关专业 
•	熟悉 Java / Spring Boot / MySQL 
•	有项目经验优先 
•	良好沟通能力 
•	能接受出差 
实现方式
优先用：
•	正则 
•	本地词表 
•	关键词短语匹配 
•	简单规则 
为什么要先规则抽
因为这些信息很多是显式写出来的，先规则抽更稳，也能减少模型负担。
________________________________________
第 3 层：标准化映射
目标
把显式抽取的原始短语映射成统一标准标签。
这一步最关键
例如：
•	熟悉 Python 
•	掌握 Python 编程 
•	Python开发能力 
都映射成：
•	Python 
再比如：
•	良好的沟通表达能力 
•	善于沟通 
•	具备较强沟通能力 
都映射成：
•	沟通能力 
建议映射的内容
•	技能标准化 
•	工具标准化 
•	专业标准化 
•	证书标准化 
•	软技能提示词标准化 
•	岗位类别归一 
•	行业标签归一 
输出中间特征
例如：
{
  "standard_job_name": "实施工程师",
  "hard_skills": ["SQL", "Linux"],
  "tool_skills": ["MySQL", "Oracle"],
  "major_requirement": ["计算机", "软件工程"],
  "degree_requirement": "本科",
  "soft_skill_hints": ["沟通能力", "学习能力"],
  "practice_requirement": ["接受出差", "有项目经验优先"]
}
________________________________________
第 4 层：岗位中间特征构建
目标
在调用大模型之前，先构建一个摘要化、标准化的中间输入。
为什么重要
不要把几十条 JD 生硬拼接后扔给模型。
应该先整理成模型更容易理解的 job_profile_input_payload。
建议这个 payload 包含
•	标准岗位名 
•	岗位类别候选 
•	高频技能 
•	高频工具 
•	学历/专业要求分布 
•	经验要求分布 
•	常见行业 
•	常见城市 
•	薪资统计 
•	软技能提示 
•	公司类型特征 
•	岗位职责摘要样本 
这一步的本质
是把“多条岗位明细”压缩成“一个岗位组摘要”。
________________________________________
第 5 层：调用统一大模型接口补全隐式岗位画像
这里才调用你的统一接口
调用方式类似：
call_llm(
    task_type="job_profile",
    input_data=job_profile_input_payload,
    context_data=...,
    student_state=None
)
让大模型做什么
模型不再负责“从零读懂所有 JD”，而是负责在中间特征基础上补充这些内容：
•	job_category 
•	job_level 
•	soft_skills 
•	suitable_student_profile 
•	core_skills 
•	plus_skills 
•	summary 
•	vertical_paths 
•	transfer_paths 
这里模型最适合做的事
•	归纳软素质要求 
•	区分核心技能与加分技能 
•	推断岗位层级 
•	补充适合学生类型 
•	推断岗位垂直路径和换岗路径 
________________________________________
第 6 层：聚合结果与输出
输出 1：业务画像结果
写入 job_profile_result
建议结构至少包含：
{
  "standard_job_name": "",
  "job_category": "",
  "job_level": "",
  "degree_requirement": "",
  "major_requirement": [],
  "experience_requirement": "",
  "hard_skills": [],
  "tool_skills": [],
  "certificate_requirement": [],
  "soft_skills": [],
  "practice_requirement": [],
  "qualification_tags": [],
  "domain_tags": [],
  "salary_summary": {},
  "city_distribution": [],
  "industry_distribution": [],
  "suitable_student_profile": "",
  "vertical_paths": [],
  "transfer_paths": [],
  "summary": ""
}
输出 2：图谱结构结果
供 Neo4j 使用，例如：
•	Job -> Skill 
•	Job -> Degree 
•	Job -> Major 
•	Job -> Industry 
•	Job -> Job (PROMOTE_TO / TRANSFER_TO) 
________________________________________
5. 这一模块建议怎么拆文件
我建议最少拆成四个文件。
1）job_profile_builder.py
职责：
•	读取岗位组数据 
•	抽显式字段 
•	做标准化映射 
•	构造中间特征 
2）job_profile_aggregator.py
职责：
•	对岗位组做统计聚合 
•	形成高频技能、学历、行业、城市、薪资摘要 
3）job_profile_service.py
职责：
•	调 builder / aggregator 
•	调统一 LLM 接口 
•	合并规则结果与模型结果 
•	输出 job_profile_result 
•	可选写回状态或保存到数据库 
4）main_job_profile_demo.py
职责：
•	用 mock 或样本数据演示整个流程 
________________________________________
6. 这一模块的设计原则
原则 1：先规则抽，再模型补
显式要求交给规则和词表，隐式岗位画像交给模型。
原则 2：先按标准岗位聚合，再画像
不要用单条岗位去代表整个岗位类型。
原则 3：输出必须面向后续匹配
这不是一个“招聘数据展示模块”，而是“岗位侧画像模块”。
原则 4：岗位画像要有“可对比字段”
后面 job_match 需要拿它和学生画像逐项对比，所以字段名和结构要稳定。
原则 5：路径结果要保守、可解释
vertical_paths 和 transfer_paths 不要让模型乱猜，前面最好先给技能重合和岗位层级提示。







一、job_match 模块的详细开发思路
1. 先明确这个模块要解决什么问题
job_match 不是简单“算一个相似度”。
它真正要做的是：
把 student_profile_result 和 job_profile_result 放在同一个评价框架里，输出一份可解释、可量化、可用于决策的人岗匹配结果。
也就是说，这一模块最终不是为了回答“像不像”，而是为了回答：
•	这个学生和目标岗位匹配度有多高 
•	匹配主要体现在哪些方面 
•	核心差距是什么 
•	哪些差距可以短期补齐 
•	是否推荐直接投递 
•	是否建议调整目标岗位 
________________________________________
2. 这个模块在你系统中的位置
建议顺序保持为：
1.	resume_parse 
2.	student_profile 
3.	job_profile 
4.	job_match 
5.	career_path_plan 
6.	career_report 
所以 job_match 是画像层和规划层之间的桥接模块。
它前面接收：
•	学生画像 
•	岗位画像 
它后面输出给：
•	职业目标设定与路径规划 
•	职业生涯报告 
________________________________________
3. 这个模块的输入来源
这一模块不要再去读原始简历、原始 JD 或清洗原表。
它只处理前面两个模块已经结构化好的结果。
输入建议包括：
学生侧
来自 student_profile_result
•	education_profile 
•	hard_skills 
•	tool_skills 
•	certificate_list 
•	qualification_tags 
•	experience_tags 
•	occupation_hints 
•	domain_tags 
•	practice_profile 
•	soft_skills 
•	potential_profile 
•	profile_completeness_score 
•	competitiveness_score 
岗位侧
来自 job_profile_result
•	standard_job_name 
•	job_category 
•	job_level 
•	degree_requirement 
•	major_requirement 
•	experience_requirement 
•	hard_skills 
•	tool_skills 
•	certificate_requirement 
•	soft_skills 
•	practice_requirement 
•	qualification_tags 
•	domain_tags 
•	suitable_student_profile 
•	vertical_paths 
•	transfer_paths 
可选输入
•	匹配维度权重配置 
•	目标城市 / 目标薪资偏好 
•	特定岗位类型的权重修正配置 
________________________________________
4. 这个模块的总体处理链
我建议你把 job_match 拆成 六层处理链。
________________________________________
第 1 层：画像读取与可比字段整理
目标
把学生画像和岗位画像中真正可以比较的字段提取出来，形成统一的比较对象。
这一步要做什么
•	读取两个结果对象 
•	检查关键字段是否存在 
•	缺失字段补默认值 
•	对字段做统一格式处理 
例如：
•	列表字段统一成 list 
•	分数字段统一成 int/float 
•	空值统一成空列表或空字符串 
输出
一个 match_input_payload，比如：
{
  "student": {
    "degree": "本科",
    "major": "软件工程",
    "hard_skills": ["Python", "SQL"],
    "tool_skills": ["MySQL", "Excel"],
    "certificate_list": ["英语六级"],
    "soft_skills": ["沟通能力", "学习能力"],
    "potential_profile": ["学习能力强"],
    "experience_tags": ["项目经历", "实习经历"]
  },
  "job": {
    "standard_job_name": "数据分析师",
    "degree_requirement": "本科",
    "major_requirement": ["计算机", "统计学", "数据科学"],
    "hard_skills": ["Python", "SQL"],
    "tool_skills": ["MySQL", "Excel", "PowerBI"],
    "certificate_requirement": [],
    "soft_skills": ["沟通能力", "逻辑分析能力"],
    "practice_requirement": ["有数据分析项目经验"]
  }
}
________________________________________
第 2 层：显式规则匹配
目标
先用程序完成可以明确比较的内容，不要一上来就交给大模型。
建议匹配内容
1）基础要求匹配
•	学历是否满足 
•	专业是否对口 
•	证书是否满足 
•	是否满足显式经验门槛 
2）职业技能匹配
•	硬技能重合度 
•	工具技能重合度 
•	核心技能缺失项 
•	加分技能命中项 
3）职业素养匹配
•	学生软技能 vs 岗位软技能要求 
•	学生实践画像 vs 岗位实践要求 
4）发展潜力匹配
•	学习能力 
•	创新能力 
•	迁移潜力 
•	学生方向提示 vs 岗位方向一致性 
为什么这一步重要
因为这一步能先产出一批很稳定的数据：
•	命中了哪些 
•	缺了哪些 
•	缺的是硬门槛还是软能力 
•	各维度的基础分是多少 
________________________________________
第 3 层：四维度评分
这一步要严格贴合 A13 的匹配要求。
建议四个维度固定为：
1）基础要求 basic_requirement_score
对比：
•	学历 
•	专业 
•	证书 
•	基本资格 
2）职业技能 vocational_skill_score
对比：
•	核心硬技能 
•	工具/框架/数据库 
•	项目技能覆盖度 
•	实习/项目与岗位技能要求的相关性 
3）职业素养 professional_quality_score
对比：
•	沟通能力 
•	团队协作 
•	抗压能力 
•	责任心 
•	实践执行力 
4）发展潜力 development_potential_score
对比：
•	学习能力 
•	创新能力 
•	成长空间 
•	岗位方向迁移潜力 
________________________________________
权重建议
你可以先给一个默认权重：
•	基础要求：25% 
•	职业技能：35% 
•	职业素养：20% 
•	发展潜力：20% 
后面允许岗位级别调整。
输出
•	四维度分数 
•	每维度命中项 
•	每维度缺失项 
________________________________________
第 4 层：语义辅助特征
这一层不是必须很复杂，但建议保留一个“轻量语义补充”的接口位。
可以做什么
•	项目经历摘要和岗位职责摘要的相似性参考 
•	学生方向 hints 和岗位类别的语义一致性参考

为什么只做辅助

所以这一层最多做：
•	平分时辅助判断 
•	解释增强 
•	结果校准参考 
不要让它主导总分。

•	软能力与潜力的解释 
•	为什么某个维度分低 
•	哪些短板可短期补齐

6. 这一模块的设计原则
原则 1：规则先打底，模型后解释
不要完全依赖模型自由打分。

因为这是项目要求的一部分。
原则 3：总分要可追溯

•	哪个维度低 
•	哪些技能没命中 
•	哪些要求满足了 
原则 4：输出必须能服务规划
这不是一个终点模块，它的结果是给后面的路径规划和报告模块直接使用的。
原则 5：解释必须面向学生
最终输出不能只有技术字段，还要有学生能看懂的结论。





一、career_path_plan 模块的详细开发思路
1. 先明确这个模块要解决什么问题
career_path_plan 不是写报告，也不是重新做人岗匹配。
它真正要做的是：
根据学生当前能力、目标岗位画像、人岗匹配结果、岗位关系路径和必要的现实信息，生成“最合适的职业目标”和“可执行的发展路径”。
也就是说，这个模块要回答：
•	这个学生当前最适合冲哪个岗位 
•	这个岗位是可以直接冲，还是需要先补齐再冲 
•	如果当前目标不够合适，有没有更稳妥的备选岗位 
•	应该走直接路径、过渡路径，还是长期成长路径 
•	短期、中期分别该做什么

4.	job_match

•	学生画像

•	SQL/统计信息（可选） 
它后面输出给：

4. 这个模块的总体处理链
我建议把 career_path_plan 拆成 六层处理链。
________________________________________
第 1 层：输入整理与决策上下文构建
目标
把学生画像、岗位画像、匹配结果和外部约束整合成一个“职业决策上下文”。
这一层要做什么
•	读取三个核心结果 
•	提取关键决策字段 
•	统一格式 
•	缺失字段补默认值 
•	形成 career_plan_input_payload 
输出示例
•	当前目标岗位 
•	当前匹配分 
•	四维度短板 
•	学生优势能力 
•	学生弱项能力 
•	可用路径候选 
•	现实约束条件 
________________________________________
第 2 层：目标岗位决策层
目标
决定“首选目标岗位”和“备选目标岗位”。
这里不要完全靠模型自由想
建议先做一层规则判断：
规则判断逻辑示例
•	overall_match_score >= 80
倾向首选目标岗位可直接投递 
•	65 <= score < 80
倾向首选目标岗位可作为主目标，但需补技能 
•	score < 65
倾向把当前岗位降为中长期目标，同时从 transfer_paths 里选一个过渡岗位 
这里的核心输出
•	primary_target_job 
•	secondary_target_jobs 
•	goal_positioning 
其中 goal_positioning 可以是：
•	可直接投递 
•	建议补齐后投递 
•	建议先从相近岗位切入 
•	建议调整目标岗位 
________________________________________
第 3 层：路径候选生成层
目标
把“首选目标”转成可执行的路径。
建议至少生成三类路径
1）直接路径 direct_path
适用于匹配度较高时
例如：
•	数据分析师 → 高级数据分析师 → BI 分析师 / 数据产品经理 
2）过渡路径 transition_path
适用于当前差距较大时
例如：
•	运营助理 → 数据运营 → 数据分析师 
3）长期路径 long_term_path
用于呈现成长上限
例如：
•	Java开发工程师 → 高级Java开发工程师 → 后端架构师 
这一步怎么做
这里建议：
•	程序先从 vertical_paths 和 transfer_paths 中取候选 
•	再让模型做筛选和解释 
而不是让模型凭空生成路径。
________________________________________
第 4 层：行动计划生成层
目标
把路径转成阶段性行动目标。
建议拆成两个阶段
短期计划 short_term_plan
1 到 3 个月，重点补齐短板：
•	补技能 
•	做项目 
•	调整简历 
•	进行岗位投递 
•	补证书或补经历 
中期计划 mid_term_plan
3 到 12 个月，重点提升竞争力：
•	争取相关实习 
•	做更高质量项目 
•	切入过渡岗位 
•	为目标岗位准备经验闭环 
输出形式建议
每项计划包含：
•	goal 
•	actions 
•	expected_outcome 
________________________________________
第 5 层：调用统一大模型接口做规划解释增强
这里调用你已经写好的统一接口：

    task_type="career_path_plan",

    context_data=rule_decision_result,

模型不应该重新做所有规则判断，而是负责：

•	fallback_strategy 
•	更自然的行动计划描述

1.	resume_parse 
2.	student_profile 
3.	job_profile 
4.	job_match

4. 这个模块的总体处理链

应该先整理成“报告编排输入”。

因为 A13 明确要求报告要具备这些部分，如果全让模型自由生成，内容容易漏项。

章节 1：学生基本情况与能力画像

•	career_path_plan_result.risk_and_gap 
•	job_match_result.improvement_suggestions 
为什么这一步很关键
因为你最终报告最好是“模型润色结构化草稿”，而不是“模型凭空写一篇作文”。
________________________________________

模型负责什么
模型最适合做：

•	不应该重新推翻前面的目标选择 
•	不应该重新决定路径 
也就是说，career_report 是编排与表达层，不是推理核心层。

输出 1：结构化报告结果

•	学生画像章节

二、这一模块适合借鉴哪些开源项目
这部分有可以借鉴的开源项目思路，但没有哪个项目需要你整套照搬。
因为开源项目里比较多的是：
•	简历分析 
•	岗位匹配 
•	报告生成框架 
•	LLM 工作流编排 
真正和你最接近的是“多阶段 AI 工作流 + 最终报告生成”。
1. LangGraph JobSearch Assistant
这个项目明确写了它用 LangGraph 来组织一个从 CV 上传到 final report generation 的模块化流程，也就是把 profile extraction、job search、最终报告编排串成一个图状工作流。(github.com)
适合借鉴什么
•	最终报告不是独立函数，而是整个多阶段流程的最后一个节点 
•	先有 extraction / analysis，再有 report generation 
•	报告生成最好建立在前序节点的结构化结果上 
对你的启发
你的 career_report 也应该是一个“汇总节点”，而不是重新做一遍分析。
________________________________________
2. peoplehub
这个项目的 README 明确提到其 AI Research Engine 支持 Comprehensive report generation with sources，并列出了像 Professional Background、Recent Projects and Achievements、Technical Expertise 这样的报告章节。(github.com)
适合借鉴什么
•	用固定章节做报告编排
